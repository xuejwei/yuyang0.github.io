1.1.1 性能

  1. 吞吐量：主要是读(QPS)和写(TPS)
  2. 延迟：

1.1.2 扩展性(scalability)

理想的情况下，分布式系统应该线性扩展，也就是说系统的计算与存储能力应该随着系统节 点数量的增多而线性的增强。为了让系统能够线性扩展，系统必须进行数据分布

1.1.3 可用性(availablity)

可用性是指系统在遇到各种异常时仍然能够正常提供服务的能力。分布式系统为了保证可用 性就必须进行容错，同时也要进行冗余，冗余又分为时间冗余和物理冗余，前者就是计算系 统中出错重试，后者就是存储系统中常用的一份数据复制三份。在存储系统中一旦使用复制， 那么就会出现一致性问题。也就是说更新数据时如何保证数据多个副本之间的一致性。

1.1.4 一致性(consistency)

因为复制的存在，所以系统就存在一致性问题，主要由这几种：

  1. 强一致性：当一个客户端对系统进行了一次更新后，那么后续的所有客户端都能读到这 次更新，比如在主从复制协议中,同步复制就是强一致的，更新必须在更新了所有副本后 才返回，这样能够保证任何一个客户端读任何一个副本时都能读到更新后的数据。
  2. 弱一致性：当一个客户端对系统进行了一次更新后，后续的客户端可能读到也可能读不 到这次更新，在常见的主从复制协议中，异步复制就是弱一致的，也就是说主副本更新 之后就返回，后台会异步的更新其他副本,极端情况可能出现丢失数据的情况
  3. 最终一致性：是弱一致性的一个特例，是指经过一段时间，客户端最终将能够读到更新后的数据，这其中就有一个不 一致窗口。

最终一致性有以下常见的变体：

  1. Causal consistency. A更新了对象之后通知B，那么B能读取A写入的值， 而与A没有因 果关系的C则可以读不到最新值。以微博为例，用户甲发布微博之后，系统通知了它的粉 丝，那么粉丝必须能看到甲发表的微博。
  2. Read your wirte consistency. 客户端总是能读取到自己的写入最新值。以博客为例， 用户必须能看到自己刚发表的博客。
  3. Session consistency. 客户端与服务器维护一个会话， 会话有效期间， 满足read your write consistency。一般在会话期间，客户端的读写都在同一台服务器上，所以 是比较容易实现的
  4. monotonic read consistency. 如果客户端已经读取了对象，那么后续读取操作不能读 到该对象的旧值。以邮件收件箱为例，用户跑到上海的时候必须也能读到他在杭州已经 读过的邮件。
  5. monotonic write consistency. 单客户端发出的写操作串行执行。

强一致性的一般实现:

  1. 使用Paxos/Raft这类一致性协议维护一个日志系统，主副本写的时候将日志提交到日志 系统，备副本根据日志系统来更新自己的状态。
  2. 主从强同步复制

弱一致性的一般实现:

  1. 单副本更新，异步复制到其它副本, 优点是不会出现更新冲突，缺点是可用性与性能不高
  2. 多副本都支持更新，优点是性能比较好，可是会出现冲突，即便引入了vector clock之 类的机制，还是会有一些冲突是无法在数据库层面解决的，必须让应用层来解决。

1.2 常用技术

1.2.1 数据分布

数据分布和分片(sharding)与分区(partitioning)表述的是同一个意思, 也就是为了扩展性， 必须将数据分布到多个节点，分布的方式主要有这几种：

  1. Round-Robin: 轮转
  2. hash分布
  3. 顺序分布

1.2.2 复制

复制是一种冗余技术，用来保证当服务器出现故障时不丢失数据，它主要用来提高系统可用 性与持久性，常见的复制协议：

  1. 主从复制: 只有主副本能用来写，主副本以同步或者异步的方式将更新同步到备副本， 备副本只能用来读
  2. NWR: W+R>N, 也就是说读的副本数加上写的副本数要大于总副本数，那么每一次读到的 副本中必然包含最近更新的数据，只是这种协议可能出现数据冲突。

1.2.3 容错

因为分布式系统大多运行于X86服务器上，X86服务器性价比高，可是不够可靠，同时数据中 心也会有各种故障，系统必须进行故障检测和故障恢复：

  1. 故障检测的常见方法有两种:
    1. 心跳: 也就是服务器定时发送心跳包，但是收不到心跳有很多情况，比如机器负载 过大，有可能是临时故障，也可能是永久故障，当机器发生故障后，我们需要迁移机 器上的服务，那么这时候必须保证不管机器是否真的挂掉，这台机器都不能在提供服 务，可是心跳检测无法保证这一点，
    2. 租约: 也就是说机器从master获得租约比如10分钟，10分钟后又申请10分钟的租约， 如果租约超时那么该节点就肯定没有提供服务了。master必须在租约上加一个提前量， 比如10分钟的租约可能需要等11分钟才能确定机器已出故障。
  2. 故障恢复：当一台机器挂掉后，系统必须增加副本，如果挂掉的机器中有主副本那么必 须为其从备副本中指定一个主副本。

1.2.4 集群管理

  1. 中心式的结构: 主要依赖中心节点来管理。
  2. P2P结构：使用gossip协议来探测机器的上线和下线。

1.3 常见理论

1.3.1 CAP

原始定义：

  • C（Consistency）：A read is guaranteed to return the most recent write for a given client
  • A（Availability）: A non-failing node will return a reasonable response within a reasonable amount of time (no error or timeout)
  • P（Partition Tolerance）：The system will continue to function when network partitions occur.

关键是这里对A的定义，它的意思是说，只要一台机器没有挂掉，那么它就应该能提供服务， 但是这只是理论上的，实际我们可以把网络不通的机器直接踢出系统，只要系统能快速的恢 复过来，那么这也是高可用的系统。

1.3.2 BASE

  1. an application works basically all the time (basically available),
  2. does not have to be consistent all the time (soft-state)
  3. but will be in some known-state state eventually (eventual consistency).

1.3.6 Vector Lock

分布式系统中由于复制的存在，所以当读的时候容易出现同一份数据的多个版本，我们必须 从这多份数据中选一份出来，一般使用的是 last write wins. 但是如何确定哪份数据是最 后写是一件很难的事，因为分布式系统没有全局的时间顺序，所以为了确定先后顺序，我们 维护一个逻辑版本，Vector Clock就是用来干这个的，可是由于分布式系统的特点，Vector CLock也只能排出partial order，而无法排出total order。也就是说系统还是可能出现冲 突，这种冲突必须在应用层面解决。

3 NOSQL

NOSQL的出现主要是为了处理以下问题才出现：

  1. 避免不必要的复杂性，关系数据库提供了很多功能，但是这些功能对于特定的应用是不 需要的。
  2. 提高系统高吞吐量
  3. 提供可用性与一致性之间的折衷，有些场景数据的一致性是不重要，那么这时候我们可 以牺牲一致性来换取性能，但是关系数据库不允许这样做
  4. 高水平扩展能力并且能运行于廉价的硬件上. 高扩展性是NOSQL的最主要的动机.
  5. 降低系统的管理与运维复杂性, 比如自动的故障检测与恢复，以及自动的负载均衡等等。

3.1.3 Redis Cluster

  1. 一致性

    redis使用的是一种弱一致性模型，在复制的时候它使用的是异步复制，所以是有可能u丢失 数据的

  2. 数据分布

    集群中有16384个slot，映射规则是直接直接CRC16然后取模(对hash tag会作特殊处理), 接 着手工将这16384个slot映射到物理机上。

  3. 复制

    redis的复制本质上是通过主从实现的，所以其实是一种基于机器的复制，对于K/V系统而言 基于slot的复制应该是一个更好的选择，基于机器的复制最大的缺点就是当一台机器挂掉后， 除非你增加一台机器，否则你没有办法增加挂掉机器上的slot的备份数目，设想有这样一个 集群： 

    A A1 A2
B B1 B2
C C1 C2

    如果slot1在A上，那么如果A1挂了，除非你添加一台机器到A这个组中否则slot1的副本数目 不可能恢复到3，当然如果你将slot1迁移到B或者C这种情况不在讨论范围之内，redis之所 以这样设计我想应该是为了简单，同时也是为了复用单机版的redis的主从同步设施, 但是 这样设计缺陷很明显，系统的自动恢复能力大打折扣，同时需要人工介入的次数增多，运维 的麻烦程度也上升了。

3.2 分布式文档数据库

3.2.1 基本原理

系统无预先定义的模式，每个document都有多个属性，更通俗的说文档就是 键值对的有序 集, 属性对应的值可以是标量值也可以是嵌入的document或者数组，系统除了可以在主键 上创建索引外，也可以在其他属性上创建索引，你不仅可以根据主键搜索，也可以根据其它 的属性来搜索。这类系统的基本术语如下：

  1. document: 类似于关系数据库的row
  2. collection： 类似于关系数据库的table
  3. db：类似于关系数据库的db

这些在不同的文档数据库可能不一样，但是他们都有等价物。

3.2.2 MongoDB

mongodb可以跑单机模式，可以跑复制集，也可以通过分片跑集群模式，下面说的都是集群 模式

  1. 数据分布

    mongodb的数据分片需要指定一个shard key，实际就是要指定以那个属性为分片的依据，分 片方式有三种：

    1. range-based sharding
    2. hash-based sharding
    3. Location-aware sharding：需要用户指定配置。

    mongodb弄了个代理性质的节点mongos，如果你以shard key来查询，那么它会直接转发对应 的分片，如果你以非shard key来查询，那么它会向所有的分片广播。

  2. 一致性模型
  3. 复制

    mongodb的复制是通过复制集来实现的，本质上是一种以机器为单位的复制。复制集可以单 独用，同时在集群环境中, 每一个分片就是一个复制集, 复制集之间数据同步是异步的, 每个复制集都有一个主节点，其它的都是备节点，主节点挂掉后，备节点会选举出一个主节 点，大多数时候程序语言的驱动会将所有的读写请求路由到主节点.

3.3 分布式表格系统

3.3.1 基本原理

表格系统实际和关系数据库类似，只是它弱化了多表关联，对事务的支持很弱(分布式事务 要用到两阶段提交协议(2PC)，性能低，而且无法容忍节点出错)，也不支持二级索引，同时 它对模式也不像关系数据库那么严格，也就是说，同一张表的不同行可以有不同的数据类型。 另外表格模型可以看做是关系模型与文档模型的混合，表格模型需要定义列族(column family)，每一个列族可以动态的添加属性。所以它不像文档模型那样完全不需要预先定义 的模式，同时也不像关系数据库那样需要非常严格的模式。

3.3.2 数据模型

表格系统提供的数据模型是一个表格，从更本质的角度讲 一个稀疏的、分布式的、持久化 存储的多维度排序 Map, 这个map的基本的K/V映射如下： 

(row:string, column:string, time:int64)->string

每一个行都有一个行健(row key), 对于列，你需要指定列族(column family), 一张表的列 不宜过多，最多几百个，但是列可以有几乎无限个。一个列使用 family:qualifier 来标 示

3.3.3 BigTable(google)/HBase

这二者非常类似, 从本质上说bigtable/HBase是建立在GFS/HDFS上的一个分布式索引，这里 以HBASE为例来说明这类系统的特点：

  1. 每一个表会按顺序切分为不同的region(子表), 每个region负责表的一个范围。
  2. 使用三种表来保存数据

    • ROOT表: 这个表只有一个region，它会保存META表的region列表，ROOT表的每一行会保 存META表的一个region的元信息，这些元信息主要是：

      • 该region所在的服务器
      • 该region服务的主键范围

      ROOT表的位置信息(元信息)保存在zookeeper中。

    • META表: 这个表用来保存用户表的所有region，以及各个region所在的RegionServer， META表的每一行都保存一个用户表的region的元信息。总之，它就是用来保存用户表 的元数据的。
    • 用户表：这个表会保存用户的实际数据
  3. RegionServer需要将每一个region的操作日志以及数据文件保存到HDFS，数据文件使用 一个或者多个store来保存，每一个store保存一个该region的列族，这也是为什么称 bigtable/hbase为列存储的原因。

4 分布式资源管理

4.1 设计动机

  1. 提高集群的资源利用率, 为了避免干扰,传统上不同的框架要部署到不同的集群, 这样就 可能造成一个集群资源紧张，一个集群的空闲资源很多，这就造成了资源浪费。分布式 的资源管理系统可以实现多个框架运行于一个集群中，提高了集群的利用效率。
  2. 服务的自动部署

4.2 发展路径

  1. 集中式：资源管理与应用管理放在一起，典型的就是MapReduce的JobTracker。
  2. 双层调度架构: 将资源管理与应用管理明确的区分开，yarn，mesos都是这类
  3. 共享状态架构；将资源调度模块简化成一些持久化的共享数据。
